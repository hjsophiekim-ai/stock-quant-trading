# 제품 아키텍처 (일반 사용자 설치형)

이 문서는 **개발자가 저장소를 clone 해서 쓰는 구조**가 아니라, **일반 사용자가 설치 파일만 받아 실행**하는 제품 기준의 최종 형태를 정의합니다.

---

## 1. 목표와 원칙

| 목표 | 설명 |
|------|------|
| 설치 후 즉시 사용 | 사용자는 **Swagger·터미널·uvicorn·`.env` 편집** 없이 앱만 실행 |
| 단일 로그인 | 이메일/비밀번호 → JWT → 이후 모든 기능은 **서버 API** 로만 |
| 브로커 비밀 분리 | 한국투자 App Key/Secret **앱 로컬 저장 금지**, **서버 암호화 저장** |
| 기본 안전 | 기본 모드 **paper**; **live** 는 서버·앱 **다중 안전장치** 없이는 잠금 |
| 크로스 클라이언트 | Windows **설치형 데스크톱**, Android **APK/AAB** 동일 백엔드 계약 사용 |

---

## 2. 제품 배포 형태 (최종)

### 2.1 Windows 데스크톱 (`.exe` / 설치 마법사)

**권장 패턴: “한 번에 설치되는 제품”**

1. **NSIS(또는 WiX) 설치 마법사** 하나로 다음을 함께 배치합니다.  
   - **Electron 셸** (`apps/desktop`)  
   - **FastAPI 백엔드 실행 파일** (예: PyInstaller 등으로 `stock-quant-backend.exe` 단일/폴더 배포)  
   - **로컬 데이터 디렉터리** (SQLite, 암호화 브로커 저장소, 런타임 상태, 포트폴리오 JSONL 등)

2. **앱 기동 시퀀스**  
   - 설치 시(또는 최초 실행 시) 백엔드를 **Windows 사용자 범위**에서 자동 기동(별도 프로세스 또는 서비스).  
   - 바인딩 주소: 기본 `127.0.0.1:8000` (고정 포트 또는 랜덤 포트 + 로컬 IPC로 URL 전달).  
   - Electron은 **`BACKEND_URL` 만 알면 됨** (이미 `runtime-config.js` 패턴 존재). 빌드 시 `http://127.0.0.1:8000` 고정이 일반 사용자에게 가장 단순.

3. **사용자에게 보이지 않는 것**  
   - Python/venv, `pip install`, `scripts/run_backend.bat`  
   - 브라우저에서 `/docs` 열기 (운영자·지원용 디버그 빌드에서만 선택 제공 가능)

### 2.2 Android (`.apk` / `.aab`)

1. **Expo / React Native** 빌드 파이프라인(EAS 등)으로 **스토어 또는 사이드로딩용 패키지** 산출.  
2. 앱에는 **`EXPO_PUBLIC_BACKEND_URL` (또는 동등 변수)만** 주입: **프로덕션 서버 HTTPS URL**.  
3. **브로커 키는 앱에 넣지 않음** — 로그인 후 **서버 API** 로 계정 등록·연결 테스트.

**백엔드 위치 선택지 (제품 정책으로 하나를 택함)**

| 모델 | 장점 | 단점 |
|------|------|------|
| **A. 데스크톱만 로컬 백엔드, 모바일은 동일 PC로 접속 불가** | PC에만 키가 있음 | 모바일 단독 사용 시 **별도 호스팅된 백엔드** 필요 |
| **B. 사용자 집 NAS/미니 PC에 백엔드 상시 기동, 모바일은 LAN/DDNS** | 한 세트의 서버 | 네트워크·SSL 설정 부담 |
| **C. 벤더 운영 클라우드 백엔드 (SaaS)** | 모바일·데스크톱 동일 URL | 운영·비용·개인정보/키 보관 정책 필요 |

제품 1.0에서는 **Windows = 로컬 번들 백엔드**, **Android = 빌드 시 주입된 고정 `BACKEND_URL`**(사용자가 설정 화면에서 **비민감** 한 URL만 변경 가능) 조합이 구현 난이도 대비 명확합니다.

### 2.3 `shared/`

- **OpenAPI 또는 수동 동기화된 TypeScript 타입 / Pydantic 스키마**로 요청·응답 모델을 맞춥니다.  
- 앱은 **민감 필드가 응답에 포함되지 않는 DTO**만 사용 (마스킹된 계좌·상태만).

## 3. 제품 기준 논리 아키텍처

```
[ 일반 사용자 ]
       │
       ▼
┌──────────────────┐     HTTPS/JWT      ┌─────────────────────────────────────┐
│ Electron (Win)   │ ◄──────────────────► │ FastAPI Backend                      │
│ Expo (Android)   │                      │ · 사용자 인증 (JWT)                  │
└──────────────────┘                      │ · 브로커 계정 암호화 저장             │
                                          │ · KIS 클라이언트 (토큰·조회·주문)    │
                                          │ · 시장국면·스크리너·신호·리스크·주문  │
                                          │ · 포트폴리오 동기화·손익 집계        │
                                          │ · paper / live 정책 게이트            │
                                          └──────────────┬──────────────────────┘
                                                         │ 모의/실전 API (서버만)
                                                         ▼
                                                [ 한국투자증권 Open API ]
```

- **앱**은 한국투자 API에 **직접 연결하지 않음**.  
- **백엔드**만 `app/clients`·`backend`의 KIS 연동 계층을 사용 (기존 AGENTS 경계 유지).

## 4. 사용자 실행 흐름 (제품)

1. **설치** — Windows: 설치 마법사 완료 후 바로가기 실행. Android: APK/AAB 설치 후 아이콘 탭.  
2. **실행** — (Win) 백엔드 프로세스가 없으면 자동 기동 → 헬스 체크 통과 후 로그인 화면.  
3. **회원가입/로그인** — 이메일·비밀번호 → `access_token` / `refresh_token` 로컬 보관(앱 스토리지).  
4. **대시보드** — JWT로 `GET /api/dashboard/summary` (또는 제품용 세분화 API) 폴링/웹소켓.  
   - 수익률, 실현/미실현, 포지션, 미체결/체결, 리스크 배너, 국면, 후보 종목 등 **한 화면 요약**.  
5. **브로커 설정** — 앱 폼 입력 → `POST /api/broker-accounts/me` (서버 암호화 저장) → `POST .../test-connection`.  
6. **Paper trading** — `POST /api/paper-trading/start` 등 **서버가 허용하는 paper 전용 API**만 사용.  
7. **Live trading** — 앱에서 “위험 확인” UI + 서버 **다중 플래그** 검증 후에만 주문 경로 개방 (기본 잠금).  
8. **성과/주문/리스크** — 별도 탭에서 동일 JWT로 포트폴리오·주문 이력·리스크 이벤트 조회 (Swagger 불필요).

---

## 5. 보안·모드 정책 (제품 관점)

- **앱 번들**: KIS App Key/Secret **미포함**.  
- **서버**: `BrokerSecretService` 등 기존 암호화 저장 경로 유지, **마스킹된 응답**만 앱으로.  
- **Live**: 서버 `TRADING_MODE`, `LIVE_TRADING*`, `LIVE_TRADING_CONFIRM`, `LIVE_TRADING_EXTRA_CONFIRM` 및 앱 측 이중 확인 UI를 **동시에** 만족할 때만 실주문 허용.  
- **기본값**: 신규 사용자·신규 설치는 **paper**.

## 6. API·화면 매핑 (제품 1.0 권장)

| 사용자 화면 | 서버 API (개념) | 비고 |
|-------------|-----------------|------|
| 로그인/가입 | `/api/auth/*` | 이미 존재 |
| 대시보드 | `/api/dashboard/summary` + 필요 시 분리 GET | 집계·폴링 |
| 성과 상세 | `/api/portfolio/*`, performance 라우트 | 동기화 선행 |
| 브로커 입력·테스트 | `/api/broker-accounts/me`, `.../test-connection` | 키는 서버만 |
| Paper 제어 | `/api/paper-trading/*`, 런타임 엔진 | KIS 모의와 데모 구분 UI 문구 |
| Live 제어 | 전용 설정 API + 서버 플래그 | 앱에서 경고·확인 다단계 |
| 리스크 | `/api/risk/status` 등 | 대시보드에 요약 |

제품 출시 전 **Swagger는 빌드 플래그로 비활성화**하거나 관리자 모드에서만 노출하는 것을 권장합니다.

## 7. 개발용 구조 → 제품용 구조로 바뀌는 점

| 구분 | 개발용 (현재 중심) | 제품용 (목표) |
|------|-------------------|----------------|
| 백엔드 실행 | 터미널, `run_backend.bat`, 수동 `.env` | 설치 시 포함, **자동 기동**, 로그는 파일·이벤트 뷰어 |
| 설정 | 루트 `.env`, 문서에 `env.paper.example` | **설치 경로** 아래 `config` 또는 최초 위저드; 민감값은 서버 DB만 |
| API 탐색 | `/docs` Swagger | 일반 사용자 **비노출**; 필요 시 지원 빌드만 |
| 백엔드 URL | 개발자가 임의 | Win: `127.0.0.1` 고정 권장; Mobile: 빌드 타입별 URL |
| 의존성 | `pip install -e .` | **패키징된 바이너리**; 사용자는 Python 미설치 |
| 모노레포 | clone 후 전체 | **릴리스 아티팩트**는 Electron 설치본 + APK/AAB + (선택) 서버 이미지 |
| 디버깅 | 로컬 로그 | 원격 로그 수집(옵션), 사용자 동의 하에 크래시 리포트 |

## 8. 구현 갭·로드맵 (코드베이스 기준 TODO)

우선순위는 **사용자가 Swagger 없이 쓸 수 있게** 만드는 것입니다.

1. **Windows**: Electron **first-run** 시 백엔드 프로세스 spawn + 종료 시 정리 + 포트 충돌 처리.  
2. **PyInstaller(또는 동등)** 로 FastAPI+uvicorn 단일 실행 진입점 (`python -m uvicorn ...` 래퍼).  
3. **프로덕션**에서 `openapi_url=None` 또는 환경 플래그로 Swagger 끄기.  
4. **모바일**: EAS 프로필로 `EXPO_PUBLIC_BACKEND_URL` 주입; 로그인 후 모든 화면이 REST만 사용하도록 정리.  
5. **Paper vs KIS 모의** 용어를 앱 UI에서 구분 (인메모리 데모 vs 실제 모의 계좌).  
6. **shared**: OpenAPI codegen 또는 수동으로 DTO 동기화 (`docs/api_contract_app.md`와 연계).

---

## 9. 요약

- **제품**은 “**앱 = 얇은 클라이언트**, **서버 = 인증·키·전략·리스크·주문·데이터**” 입니다.  
- **Windows**는 **백엔드를 설치본에 묶어 자동 실행**하는 것이 일반 사용자 경험에 가장 맞습니다.  
- **Android**는 **반드시 도달 가능한 백엔드 URL**이 필요하므로, 로컬 PC 전용이면 **동일 사용자의 모바일 시나리오**를 제품 정책으로 명시하거나 **호스팅 백엔드**를 제공해야 합니다.  
- 개발 모노레포 구조는 유지하되, **배포 파이프라인·런처·비-Swagger UX**가 제품 완성도를 결정합니다.
